# 军备系统自洽性审查

> 审查日期：2026-06-09　分支：`feat/trigger-gating`
> 审查范围：军备生产 → 库存 → 调拨 → 军队装备 → 军队升级 → 单价军费 全链路
> 结论一句话：**实物军备链（产→存→拨）完全闭环且自洽；但「军队升级」是半成品（设定 v3 已铺，代码没接），且有几处设定字段无人消费。**

---

## 一、数据结构总览

军备系统由 **4 张表 + armies 表 1 列** 组成：

| 对象 | 位置 | 含义 |
|---|---|---|
| `weapons` | `db/schema.py:284` | 武器型号注册表：id/name/tier/power/cost/equip_per_unit/requires_tech |
| `arms_stock` | `db/schema.py:297` | 中央总库库存：weapon_id → qty（单主键，不重复） |
| `army_arms` | `db/schema.py:305` | 各军持有明细：(army_id, weapon_id) → qty |
| `arms_logs` | `db/schema.py:316` | 军备流水：army_id=NULL 记总库变更，非 NULL 记某军变更（双向记账） |
| `armies.equipment` | `db/schema.py:208` | 该军「装备」评分 0–100（盘面字段，喂 LLM） |

设定来源：`content/weapons.json`（型号表 + 开局库存）、`content/troop_cost.json`（兵种/军需分层表）。

## 二、逐环节自洽性结论

### ✅ 环节 1：军备生产 —— 闭环

- **位置**：`flows.py:apply_fixed_period_flows`（每月固定财政 tick 内）。
- **机制**：建筑的 `output_metric` 命中武器型号 id → 调 `db.add_arms_stock()` 月末自动入总库。
- **当前配置**：京营火器局月产火铳 440、定海卫炮台月产佛郎机 2（受 `requires_tech` 前置科技门控）。
- **验证**：`tests/test_arms_and_troops.py` —— 开局火铳 1200 + 月产 440 = 1640 ✓。
- **判定**：生产全自动、无人工干预、科技锁完整。**自洽**。

### ✅ 环节 2：军备库存 —— 闭环

- 总库 `arms_stock`、各军 `army_arms`、流水 `arms_logs` 三表分工清晰，双向记账。
- 型号自动注册：`_ensure_weapon_registered()`（`db/arms.py`）支持 seed 与 runtime 两种入库；LLM 推演出未列型号按 tier 关键词归档、给默认属性动态注册。
- 增减入口：`add_arms_stock()`（生产/缴获/销毁）、`apply_arms_stock_deltas()`（extractor 落库）。
- **判定**：结构完整、记账闭环。**自洽**。

### ✅ 环节 3：军备调拨（拨发到军）—— 闭环

- **触发**：兵部/工部 court tool `dispatch_arms` → `session.py:740` 截获 → `db.apply_arms_dispatch()`（`db/arms.py:154`）。
- **5 步原子流程**（`db/arms.py:154-209`）：
  1. 校验军队存在；
  2. **硬卡库存** `actual = min(请拨, 库存)`，库不足照实少发并提示；
  3. 扣总库 `arms_stock.qty -= actual` + 写流水；
  4. 增 `army_arms.qty += actual` + 写流水；
  5. 提该军 `equipment`：`raw_gain = actual × equip_per_unit ÷ (manpower/1万)`，钳 0–100。
- **判定**：校验→扣库→增军→提评分，流水双向，**绝不凭空生成、绝不超发**。**自洽**。

> 注意：当前**没有「军队→总库」回收**和**「军队↔军队」横向调拨**，只有「总库→军」单向拨发。这是设计取舍（符合史实：械由中枢统调），非 bug，但若想做战损回收/转隶需另补。

### ✅ 环节 6：兵种 → 单价 → 军费 —— 闭环

- **位置**：`content.py:595-813`（`troop_maintenance_total` / `troop_maintenance_delta`）。
- **三场景全自动推导**：
  - 开局：按 `troop_composition` 各兵种 manpower × per_kilo 求和 → `maintenance_per_turn`；
  - 扩编：`armies.py:324`，改 manpower 未显式给 maintenance 时按 delta 自动补军费；
  - 改编制：`armies.py:386`，改 `troop_composition` 自动整体重算军费。
- **验证**：`test_arms_and_troops.py` 断言 `maintenance == troop_maintenance_total(composition)` ✓。
- **判定**：杜绝「人数涨、军费不涨」。**自洽**。

## 三、不自洽 / 断裂点（需决策）

### ❌ 断裂点 A：军队升级（兵种进阶）—— 设定铺了，代码完全没接

这是**唯一真正的功能性断裂**。

- `content/troop_cost.json` 已升到 **v3**，为每个 tier 定义了 4 个新字段：
  `category`（步兵/骑兵/海军/空军）、`equipment`（依赖军备）、`upgrades`（可升级到的兵种）、`locked`（是否锁定）。
  并铺了完整升级线（步兵→线列→散兵→班组→机械化；桨帆→风帆→铁甲→无畏舰；侦察机→战斗机/轰炸机 等）。
- **但代码端 `content.py:509-539` 的加载器只读 `tier / per_kilo / keywords` 三个字段，其余四个（category/equipment/upgrades/locked）被静默丢弃。** 全仓 grep 确认：
  - 无 `upgrade_troop` 之类逻辑；
  - extractor 无 `troop_upgrades` 字段；
  - simulator 不产升级建议；
  - 兵种只能被整体改写（`troop_composition`），不能按升级线渐进进阶。
- **后果**：玩家/LLM 永远走不到这些 tier；`upgrades/locked/equipment/category` 是纯死数据。这是 Victoria 3 式科技兵种树的「半成品」状态——设定先行，循环未同步。

**需要决策（二选一）**：
1. **回退**：把 troop_cost.json 降回 v1 风格（去掉 upgrades/locked/category/equipment），只留实际生效的 tier/per_kilo/keywords，避免设定撒谎；
2. **补齐**：实现升级闭环——加 court tool/extractor 字段 `troop_upgrades`，校验 `upgrades` 合法目标、`locked` 门控（依赖科技/军备）、`equipment` 需求，落库改 troop_composition 并重算军费。

### ⚠️ 断裂点 B：`weapons.json` 的 `power`（战力权重）字段无人消费

- `power` 一路存进 `weapons` 表（`db/arms.py:38/44/78`），但**没有任何代码读它做计算**。
- 注释说是「战力权重」，实际既不进战力公式（项目无战力公式，见下），也不喂 LLM 排序。
- **判定**：当前是装饰性字段。要么删，要么在 simulator payload 里把它喂给 LLM 当定性参考。

### ⚠️ 断裂点 C：开局库存未分配到军

- `weapons.json.opening_stock`（火铳 1200、鸟铳 300…）seed 时全进**总库** `arms_stock`，`army_arms` 开局为空。
- 即开局所有械都在中枢、没下发到具体军队。需皇帝/兵部显式 `dispatch_arms` 才到军。
- **判定**：符合「中枢统调」设计，但若期望开局各军自带装备，需在 seed 时按 armies 的 equipment 反推一份初始 army_arms。属设计取舍，非 bug。

## 四、一个重要澄清：本项目「没有程序化战力计算」

探查中一度怀疑「equipment 评分是否真影响战力」。核实结论：

- **`equipment / morale / training / supply / mobility` 等全是「盘面字段」**，由 `simulation.py` 塞进 simulator agent 的 user payload（`simulation.py:439/632`），**战力强弱由 LLM 在月末邸报里定性叙事判定，全仓没有任何战力公式**。
- 这是本项目的核心设计（数值由 LLM 评判，代码不重复算），**符合 CLAUDE.md「LLM 已判过的事代码端不重复校验」**。
- 所以「equipment 被更新后是否影响战力」这个问题**不成立**——它影响的是 LLM 看到的盘面，进而影响叙事，这就是它的全部作用，并无断裂。

## 六、建议优先级

1. **先决策军队升级（断裂点 A）**：这是唯一影响玩法完整性的真断裂。要么回退 v3 设定，要么补 `troop_upgrades` 闭环。**建议补齐**，因为 v3 兵种树是已投入的设计资产。
2. **清理 `weapons.power`（B）**：顺手删或喂 LLM，避免死字段误导后续维护。
3. 开局库存分配、军↔军调拨（C）：按玩法需要再定，当前不阻塞。
